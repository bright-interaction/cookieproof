Setup Guide

1
Publish Your Configuration

Configure your banner in the Editor tab — choose colors, integrations, texts, and categories. Then go to Script → Publish to Server to save your config. The loader script fetches it automatically, so any changes you publish go live within 5 minutes.

2
Add One Script Tag

Paste this single line into your website. It loads CookieProof and your published configuration automatically:

<script src="https://consent.brightinteraction.com/loader.js" data-domain="yourdomain.com"></script>

Add the script tag inside <head> or before </body>:

  1. Copy the script tag above
  2. Open your HTML file
  3. Paste it inside <head> (recommended) or before </body>
  1. Install the WPCode plugin (or Insert Headers and Footers)
  2. Go to Code Snippets → Header & Footer
  3. Paste the script tag in the Header section
  4. Save changes

Alternatively, paste it directly in your theme's header.php before </head>. Use a child theme to avoid losing changes on updates.

  1. Open your Webflow project
  2. Go to Project Settings → Custom Code
  3. Paste the script tag in the Head Code section
  4. Publish your site

For React or Next.js, add the loader to your root layout:

// Next.js (app/layout.tsx)
import Script from 'next/script'

export default function Layout({ children }) {
  return (
    <html>
      <head>
        <Script
          src="https://consent.brightinteraction.com/loader.js"
          data-domain="yourdomain.com"
          strategy="afterInteractive"
        />
      </head>
      <body>{children}</body>
    </html>
  )
}

In your Astro layout file (e.g. Layout.astro), use the is:inline directive to prevent Astro from bundling the script:

<!-- src/layouts/Layout.astro -->
<html>
  <head>
    <script is:inline
      src="https://consent.brightinteraction.com/loader.js"
      data-domain="yourdomain.com"></script>
  </head>
  <body>
    <slot />
  </body>
</html>
The is:inline attribute is required — without it, Astro will try to bundle the external script and it won't work.
3
Update CSP Headers (if applicable)

If your site sets a Content-Security-Policy header, you need to whitelist CookieProof. Add https://consent.brightinteraction.com to these directives:

script-src 'self' https://consent.brightinteraction.com ...; connect-src 'self' https://consent.brightinteraction.com ...;

The script-src directive allows loading the loader and UMD bundle. The connect-src directive allows fetching the configuration and submitting consent proofs.

Skip this step if your site doesn't set a CSP header. Most sites don't — this is only needed for sites with explicit security policies.
4
Gate Third-Party Scripts

To block analytics and marketing scripts until consent is granted, change their type to text/plain and add a data-consent attribute with the matching category:

<!-- Before: fires immediately -->
<script async src="https://www.googletagmanager.com/gtag/js?id=G-XXXXX"></script>

<!-- After: fires only when analytics consent is granted -->
<script type="text/plain" data-consent="analytics"
        src="https://www.googletagmanager.com/gtag/js?id=G-XXXXX"></script>

Available categories: analytics, marketing, preferences. The necessary category is always active.

If you configured integrations in the Editor, the full embed snippet (below) includes these gated scripts automatically.
5
Verify

After deploying, run through this checklist:

  • Open your site — the consent banner should appear
  • Accept cookies — the banner dismisses and gated scripts fire
  • Check the History tab — a consent proof should appear
  • Reject cookies — gated scripts should NOT fire (verify in DevTools → Network)
  • Refresh the page — the banner should NOT reappear (consent is remembered)
  • Clear site data — the banner should reappear

Banner Documentation

Everything you need to know about the consent banner that runs on your website. This covers the banner itself — configuration, styling, events, and how it handles consent. For server-side endpoints, see the API tab.

How It Works

The consent banner is a lightweight web component (<cookie-consent>) that you add to your website. Here's what happens when a visitor loads your page:

  1. The loader script fetches your saved configuration from the server (~0.8 KB)
  2. The banner appears asking the visitor to accept or reject cookie categories
  3. Their choice is saved locally (in localStorage or a cookie) so they aren't asked again
  4. Scripts are unblocked — only the categories the visitor accepted get activated
  5. A consent proof is sent to the server for your audit trail

The entire banner is ~22 KB over the wire (gzipped). It has no dependencies and works on every modern browser.

Categories

Consent is organized into categories. Each category groups related cookies and scripts together, and visitors can accept or reject them independently.

The Four Default Categories

CategoryAlways On?What It Covers
necessaryYesEssential cookies your site needs to function (login sessions, security tokens, shopping carts). Visitors cannot turn these off.
analyticsNoTracking tools like Google Analytics, Hotjar, or Plausible. Helps you understand how people use your site.
marketingNoAdvertising pixels from Meta, LinkedIn, TikTok, etc. Used for retargeting and ad measurement.
preferencesNoCookies that remember user choices like language, theme, or layout preferences.

Cookie Declarations

You can list the specific cookies each category uses. These appear in the preferences panel so visitors can see exactly what they're consenting to:

categories: [{
  id: 'analytics',
  declarations: [
    { name: '_ga', provider: 'Google', purpose: 'Distinguishes unique users', expiry: '2 years' },
    { name: '_gid', provider: 'Google', purpose: 'Session tracking', expiry: '24 hours' },
  ]
}]
Script Gating

Script gating prevents third-party scripts from running until the visitor gives consent. This is the core of GDPR compliance — no tracking before permission.

How to Block a Script

Change type="text/javascript" to type="text/plain" and add data-consent with the category name. CookieProof will convert it back to a real script once that category is accepted.

<!-- This Google Analytics script won't run until "analytics" is accepted -->
<script type="text/plain" data-consent="analytics"
        src="https://www.googletagmanager.com/gtag/js?id=G-XXXXXX"></script>

<!-- Inline scripts work the same way -->
<script type="text/plain" data-consent="marketing">
  fbq('init', '1234567890');
  fbq('track', 'PageView');
</script>

Iframes, Images & Stylesheets

For non-script elements, use data-src instead of src (or data-href for links):

<!-- YouTube embed blocked until marketing consent -->
<iframe data-consent="marketing"
        data-src="https://www.youtube.com/embed/VIDEO_ID"></iframe>

<!-- Tracking pixel blocked until marketing consent -->
<img data-consent="marketing"
     data-src="https://example.com/pixel.gif" style="display:none;" />

<!-- External font blocked until preferences consent -->
<link data-consent="preferences"
      data-href="https://fonts.googleapis.com/css2?family=Inter"
      rel="stylesheet" />

The engine also watches for dynamically-added elements, so scripts injected by other code are caught too.

Events

The banner fires events when consent changes. Use these to trigger your own logic — like initializing analytics only after the visitor says "yes".

const cc = document.querySelector('cookie-consent');

// Listen for any consent change
const unsub = cc.on('consent:update', (detail) => {
  console.log(detail.consent);   // { version, timestamp, categories, method }
  console.log(detail.changed);   // ['analytics']  — which categories changed
});

// Stop listening
unsub();
EventWhen It Fires
consent:initBanner initialized. If the visitor already made a choice before, it's loaded from storage.
consent:updateAny time consent changes — accept, reject, or custom selection.
consent:accept-allVisitor clicked "Accept All".
consent:reject-allVisitor clicked "Reject All".
consent:gpcBrowser sent a Global Privacy Control signal and non-essential cookies were auto-rejected.
consent:expiringConsent is about to expire. Includes daysRemaining in the event detail.
consent:category:*A specific category changed. Example: consent:category:analytics fires when analytics consent is granted or revoked.

Dark Mode

Set theme: 'auto' (the default) and the banner automatically matches the visitor's system preference. You can also force it with theme: 'dark' or theme: 'light'. Dark mode flips all colors automatically — no extra CSS needed.

Google Consent Mode V2

If you use Google Analytics, Google Ads, or any Google tag, you need Consent Mode V2. It tells Google what the visitor has agreed to.

Enable it with one setting: gcmEnabled: true. The banner then:

  1. Sets all Google signals to denied when the page loads (before the visitor chooses)
  2. Updates the signals to granted as the visitor accepts categories

Which Categories Map to Which Signals

CategoryGoogle Signals That Get Granted
analyticsanalytics_storage
marketingad_storage, ad_user_data, ad_personalization
preferencesfunctionality_storage, personalization_storage

security_storage is always granted — it falls under "necessary".

Custom Mapping

If you need a different mapping (for example, if you want analytics to also grant functionality_storage), you can override it:

cc.configure({
  gcmEnabled: true,
  gcmMapping: {
    analytics: ['analytics_storage', 'functionality_storage'],
    marketing: ['ad_storage', 'ad_user_data', 'ad_personalization'],
  }
});
Global Privacy Control

Some browsers (like Firefox and Brave) send a Global Privacy Control signal that says "this person doesn't want to be tracked." When this is detected, the banner:

  1. Automatically rejects all non-essential categories
  2. Saves the choice with method: 'gpc'
  3. Fires the consent:gpc event
  4. Skips showing the banner (the choice is already made)
  5. Shows the floating trigger button so the visitor can still change their mind

This is enabled by default (respectGPC: true). You can turn it off with respectGPC: false if GPC doesn't apply in your region.

API Reference

The CookieProof API lets you manage banner configurations, read consent proofs, and build integrations programmatically. For banner-specific documentation (how the widget works on your website), see the Docs tab.

Authentication

All authenticated endpoints use JWT tokens passed as a Bearer token. Tokens are scoped to your organization and user.

Browser Sessions (httpOnly Cookies)

For browser-based access (like this configurator), login sets a secure httpOnly cookie automatically. The cookie includes CSRF protection:

curl -X POST https://your-server/api/auth/login \
  -H "Content-Type: application/json" \
  -c cookies.txt \
  -d '{"email": "you@example.com", "password": "your-password"}'

# Response includes user info and CSRF token:
# { "user": {...}, "org": {...}, "csrf_token": "abc123..." }
# The JWT is set as httpOnly cookie (not in response body)

For state-changing requests (POST, PUT, DELETE), include the CSRF token in the X-CSRF-Token header.

API Key Authentication

For server-to-server integrations, use an API key in the Authorization header:

Authorization: Bearer your-api-key-here

Generate API keys from Settings → API. Keys are scoped to your organization.

Public Endpoints (no auth needed)

Two endpoints are intentionally public because they're called from visitors' browsers:

  • GET /api/config/:domain — fetches the banner configuration (called by the loader script)
  • POST /api/proof — records consent choices (called by the banner via sendBeacon)

Allowed Domains

Allowed domains control which websites can submit consent proofs to your server. This prevents other sites from writing to your audit trail. The banner loader also uses this to validate the origin.

GET /api/settings/domains AUTH

List all allowed origins. Each entry shows whether it comes from the server environment (env) or was added through the dashboard (database).

// Response
{
  "domains": [
    { "origin": "https://mysite.com", "source": "database", "created_at": 1708819200000 },
    { "origin": "https://staging.mysite.com", "source": "env", "created_at": null }
  ]
}
POST /api/settings/domains AUTH

Add an allowed origin. Must be a full HTTPS URL (e.g. https://mysite.com).

// Request body
{ "origin": "https://mysite.com" }
DELETE /api/settings/domains/:id AUTH

Remove a domain from the allow list. Only works for domains added through the dashboard — environment-configured domains cannot be removed via the API.

Rate Limits

The API applies rate limits to prevent abuse. Authentication endpoints are limited per IP address; other endpoints may be per-user.

EndpointLimit
POST /api/proofGeneral rate limit (high ceiling for legitimate traffic)
POST /api/auth/login5 attempts per 15 minutes per IP
POST /api/auth/register5 attempts per 15 minutes per IP
POST /api/auth/change-password5 attempts per 15 minutes per IP
GET /api/config/:domain100 requests per hour per IP
GET /api/proof/export10 exports per hour per user
GET /api/scan10 per hour per IP
POST /api/agency/report/send10 per hour per IP
*/api/webhooks/*30 operations per hour per user

When rate limited, the API returns HTTP 429 with a descriptive error message.

Two-Factor Authentication Lockout

After 5 failed 2FA code attempts, the account is locked for 15 minutes. This protects against brute-force attacks on TOTP codes.
